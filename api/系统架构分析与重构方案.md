# 系统架构分析与高性能重构方案

> **约束**：现有数据库表结构保持不动，仅重构应用层架构
> **目标**：建立标准化、可扩展的高性能架构框架，适用于大用户量级
> **起步硬件**：2 核 8G（可水平扩展，加服务器走集群）
> **思路**：只做好 2~3 张表作为标准模板，验证通过后由生成器批量产出

---

## 技术选型对比

| 层 | 旧方案 | 新方案 | 提升理由 |
|---|--------|--------|---------|
| HTTP 框架 | Koa 2 (~33k req/s) | **Fastify 4** (~77k req/s) | 2.3x 性能，内置 JSON 序列化、schema 校验 |
| 微服务 | Moleculer (broker.call) | **去掉，Fastify 插件** | 本地部署 broker.call 纯增加 5ms 开销 |
| 日志 | console.log (同步阻塞) | **pino** (Fastify 内置) | 异步、零配置、最快的 Node.js 日志库 |
| 缓存 | ioredis (未启用) | **ioredis + lru-cache** | L1 进程内 + L2 Redis 二级缓存 |
| 限流 | 无 | **@fastify/rate-limit** | 插件式集成，IP 维度 200 req/min |
| ORM | Sequelize (raw SQL) | **保留 Sequelize** | raw SQL 性能好，baseSql JOIN 模式成熟 |
| 进程管理 | 无 | **PM2 cluster** | 单机多进程 → 多机集群 |

### 为什么去掉 Moleculer？

Moleculer 设计用于分布式微服务，但你的系统是单体部署：
- `broker.call()` 每次调用 ~5ms 序列化/反序列化开销
- 每个请求都经过 broker 路由，增加不必要的中间层
- Fastify 的插件（`app.register`）提供同样的模块隔离能力
- 去掉后：请求 → 路由 → handler → 数据库，链路最短

如果未来需要分布式，加一层 API Gateway 转发即可，不影响业务代码。

## 核心优化

### 1. 消灭 N+1（baseSql JOIN）

旧版 `biz_customer.getList` = 1 主查询 + 4 次 `setValues`（user/product/repay/status）= **5~8 条 SQL**

新版 baseSql 直接 JOIN：

```sql
select biz_customer.*,
       _user.title as user_id_title,
       _product.title as product_id_title,
       _repay.title as repayment_status_title,
       _status.title as status_title
from biz_customer
left join gb_user _user on _user.id = biz_customer.user_id
left join biz_product _product on _product.id = biz_customer.product_id
left join biz_repayment_status _repay on _repay.id = biz_customer.repayment_status
left join biz_customer_status _status on _status.id = biz_customer.status
where biz_customer.id > 0 {0} {1}
order by {2}
```

**结果：1 条 SQL 搞定列表 + 所有关联标题。**

### 2. DTO 裁剪

列表不再传 `action_data`/`ocr_data`/`user_data`/`images` 等 JSON 大字段，传输量减少 80%+。

### 3. 二级缓存

```
请求 → L1 lru-cache (进程内, ~0.1ms) → L2 Redis (~1ms) → L3 MySQL
```

### 4. 参数化查询 (safeWhere)

```javascript
const w = this.safeWhere();
w.like('biz_customer.title', keyword, 'or');
w.eq('biz_customer.status', status);
w.in('biz_customer.product_id', [1, 2, 3]);
const { sql, params } = w.build();
```

所有查询条件自动参数化，杜绝 SQL 注入。

## 性能预估

| 指标 | 旧架构 (Koa+Moleculer) | 新架构 (Fastify) | 多机集群 |
|------|----------------------|-----------------|---------|
| HTTP 框架性能 | ~33k req/s | ~77k req/s | ×N |
| 单请求 SQL 数 | 8~20 | 1~2 | 1~2 |
| broker.call 开销 | ~5ms/次 | 0ms | 0ms |
| 平均响应时间 | 200~500ms | 20~80ms | 15~50ms |
| QPS 上限 (2核) | ~100 | ~1000~1500 | ×N |
| 缓存命中率 | 0% | 80%+ | 80%+ |

> 10 万日活所需 QPS ≈ 300~500，单机即有 2~3 倍余量。

## 扩展路径

```
阶段一（当前）：单机 2 核 8G
  → PM2 × 2 进程, 连接池 50/进程
  → QPS ~1000~1500

阶段二：加服务器
  → Nginx 负载均衡 → 2+ 台机器
  → MySQL 主从读写分离
  → QPS ~3000+

阶段三：容器化
  → Docker / K8s 自动伸缩
  → Redis Cluster
  → QPS ~10000+
```

应用层代码三个阶段完全不改，只改部署配置。
